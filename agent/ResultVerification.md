# 结果验证准则

## 1) 测试准则(必须遵守)
1. 对每个要开发的需求, 先理解需求核心、用户场景、成功标准与主要风险, 再自行判断是否需要补充测试, 不允许机械地“逢需求必堆测试”.
2. 开发阶段必须显式判断 `Unit Test`、`UI Test`、`AI 长逻辑测试` 是否需要；选择顺序固定为：先 `Unit Test`, 再 `UI Test`, 最后才是 `AI 长逻辑测试`. 详细定义见 `docs/test/README.md`.
3. `Unit Test` 默认由你直接补充，优先覆盖真实业务规则、关键状态流转、异常路径与高回归风险点, 不写仅为了测试而存在的低价值用例.
4. 只要需求涉及用户交互、关键业务流程或跨页面/跨状态操作, AI 除单元测试外, 还需要先整理多个“模拟用户操作的 UI 测试路径”, 明确前置条件、关键输入、操作步骤、预期结果与覆盖目的；经用户确认后再编写模拟器 UI 流程测试.
5. `AI 长逻辑测试` 仅当前两类测试难以覆盖时启用；具体触发条件见 `docs/test/README.md` 的“AI 长逻辑测试：何时启用”. 启用前必须先写清目标、前置条件、为什么不能只靠 `Unit Test` / `UI Test`, 执行边界、成功标准与证据要求，并使用 `docs/test/templates/ai-long-flow-test-template.md`. 禁止把“让 agent 自己试试”当作测试方案.
6. 在测试执行中新增的坑点、工具问题、环境问题、有效策略，应沉淀到 `docs/test/pitfalls.md`, 执行测试前主动读取文件以便快速规避风险.
7. 若 AI 长逻辑测试涉及模拟器、浏览器或长时运行代理，仍须遵守“串行、可观察、遇阻及时上报”.


## 2) iOS模拟器使用原则

1. 单个会话内禁止并行测试, 禁止任何导致并行运行模拟器的场景, 只允许串行测试.
2. 进行UI测试时, 模拟器冷启动时比较缓慢, 需要多等待一些时间. 最长不超过2分钟.
3. 模拟器只使用 iPhone 17 Pro.
