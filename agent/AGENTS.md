# AGENTS 协作规范

## 0) 执行起点

- 先读 `SOUL.md`, 在读本准则.
- 读取`ResultVerification`来了解任务或需求的验证准则.
- 开始新一轮工作前，优先阅读 `docs/llm_ctx/active-tasks.md`了解目前任务.
- 动手前先确认真实目标、用户场景、成功标准、主要风险与边界归属.
- 如果用户给出的实现方式不是最优解，应先提出更合适的方案与理由，而不是直接照抄需求表述.

## 1) 沟通与基本原则

1. 注释、分析、回复统一使用中文，技术术语可保留英文.
2. 正常情况下优先复用现有能力：`MCP` > `skills` > 临时脚本.
3. 读取遵循最小必要原则，避免无差别全量扫描.

## 2) 代码生成原则

1. 每个独立的功能块, 函数, 以及UI层级, 需要增加中文注释.
2. 当需求的实现, 需要兼容旧逻辑时, 优先反问人类是否需要兼容.

## 3) 文档查询与记录(按需使用)

1. 需求文档维护在`docs/prd/`目录下(包含每个版本的需求文档).
2. 技术文档维护在`docs/dev/`目录下(包含技术架构, 运维命令).
3. 每个版本需求开发的`阶段性`计划和`全阶段`计划维护在`docs/plans/`目录下(了解全部开发计划以及确认下一阶段需求时使用).
4. 项目概览维护在`docs/overview/`目录下(快速了解项目内容和结构时使用).
5. 开发标准和实现计划必须位于 `docs/superpowers`目录下(了解某个具体需求的开发标准和实现时使用), 使用`superpowers`这个skill时需遵守.

## 4) 交付输出规范

1. 明确说明：改了什么、为什么改、影响哪些文件
2. 明确说明：执行了哪些验证（编译/运行/测试）以及结果.
3. 明确说明：未执行项及原因（工具缺失、设备不可用、权限限制等）.

## 5) 其他补充(此项目特别注意)
1. 此项目分为iOS端(`LocafAppForiOS`)和服务端`LocafServer`.
2. 此项目的iOS端的UX高保真设计稿, 位于`realHtml/html`, 只读取`index.html`来了解页面.
